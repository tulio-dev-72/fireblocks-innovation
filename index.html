How AI meets custody

Fireblocks developer tooling: SDK, CLI & MCP

Fireblocks exposes three layers to builders. This suite leans on all three: the official SDK as the backbone, CLI-style scripts to run the sandbox, and a custom MCP server that lets AI agents operate on custody safely.

SDK

The primary developer interface for building on Fireblocks. Create vault accounts and wallets, transfer assets, approve transactions, and manage policy. Requests authenticate with an API key plus an RSA private key.

TypeScript Python Java Go C#
In this suite: @fireblocks/ts-sdk (v7, v12, v19) across four of the five apps. API docs →

CLI

Command-line tooling to interact with the platform without a full application: test API calls, generate signatures, validate connectivity, and run automation. Many teams build their own CLI wrappers around the SDK.

auth checks signatures connectivity automation
In this suite: custom Node scripts that verify auth and seed the sandbox with vaults and test assets. CLI docs →

MCP

Model Context Protocol lets AI assistants like Cursor, Claude, and ChatGPT securely operate on Fireblocks resources. The assistant talks to an MCP server, and the MCP server uses the Fireblocks SDK underneath.

Cursor Claude ChatGPT agents
In this suite: a custom JSON-RPC MCP server exposing read-only Fireblocks tools (list vaults, get transactions) to agents. Building with AI →

Constraints & honesty

Engineering within the Fireblocks sandbox

Everything here runs against one shared sandbox workspace on public testnets. That keeps it real and safe, but the sandbox has genuine constraints. Here is how they were navigated.

01
Testnet assets, acquired by hand
Every balance is testnet value with no real-world worth. Funding vaults meant generating Fireblocks deposit addresses, pulling from public faucets (Polygon Amoy POL, Ethereum Sepolia ETH), and minting sandbox test tokens. A slow, rate-limited loop, automated with seed scripts.
02
Five API users, allocated carefully
The sandbox caps API users at five. With five apps sharing one workspace, API users were inventoried and assigned deliberately, retiring unused ones to make room.
03
No custom TAP authoring
The sandbox does not permit authoring restrictive Transaction Authorization Policy rules. Where a demo needs a hard approval gate, it is enforced at the application layer and labeled as such; the live TAP is surfaced read-only.
04
Mobile co-signer enrollment
Publishing policy changes and approving held transactions expects an owner or co-signer enrolled on the Fireblocks mobile app via device QR, which an API-only sandbox cannot fully provide.
05
Faucet and asset coverage
Testnet faucet liquidity and supported-asset coverage are limited, so scenarios were built around assets the sandbox actually supports (for example, Sepolia ETH standing in for a production Polygon leg).
06
Rate limits and latency
Sandbox API rate limits and variable latency shaped polling intervals, webhook handling, and how aggressively each app refreshes live data.
